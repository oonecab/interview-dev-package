# 软件开发工程师 · Take-home 实战开发题

**题目**：智慧空间设备监控与 AI 工单助手

**提交时限**：收到题目后 3 个工作日内提交

---

## 1. 业务背景

"星汇智慧产业园" 拥有 5 栋楼宇、约 150 台受监控设备（电梯、空调、水泵、照明、消防等）。运维团队需要一个 Web 管理系统来实时查看设备状态、处理告警、管理维修工单，并通过一个 AI 助手用自然语言快速查询设备信息和创建工单。

---

## 2. 技术栈要求

| 项目 | 要求 |
|---|---|
| 前端框架 | React 18+ / TypeScript（**必须**） |
| UI 组件库 | 不限（Ant Design / MUI / shadcn/ui / 自写均可） |
| 状态管理 | 不限（选你认为合适的方案，在设计说明里解释为什么） |
| 构建工具 | Vite（推荐） |
| 包管理器 | 不限（pnpm 推荐） |
| Mock API | 附件提供了开箱即用的 mock server（见下方"Mock Server 使用"） |
| Node.js | 18+（运行 mock server） |

---

## 3. Mock Server 使用

附件 `mock-server/` 目录包含一个预配置的 Express mock API，提供设备、告警、工单和 AI 对话全部接口。

```bash
cd attachments/mock-server
pnpm install     # 或 npm install
pnpm start       # 或 node server.js
# API 运行在 http://localhost:3001
```

API 接口说明见 `api-spec.md`。**你不需要修改 mock server**，但如果你认为接口不满足需求，可以扩展——在设计说明里说明你改了什么。

## 4. 功能需求

### 模块 A：设备状态看板（必做）

1. **楼栋选择器**（左侧边栏）：点击切换楼栋，设备列表随之更新
2. **设备列表**：展示设备名称、类型、楼栋/楼层、状态（正常/告警/故障/离线）
3. **状态筛选**：可按状态和设备类型过滤
4. **设备详情**：点击设备弹出详情（弹窗或抽屉），展示最近的告警记录
5. **统计概览**：顶部展示各类型设备数量和各状态计数

### 模块 B：工单管理（必做）

1. **工单列表**：展示工单编号、标题、状态、优先级、创建时间
2. **状态筛选**：可按工单状态（待派单/已派单/处理中/已完成）过滤
3. **创建工单**：表单包含：标题、描述、关联设备（下拉选择）、优先级
4. **状态流转**：支持在工单详情里点击按钮推进状态（待派单 → 已派单 → 处理中 → 已完成）

### 模块 C：AI 助手对话（必做）

1. **聊天界面**：右侧边栏，支持用户输入文本、展示对话历史
2. **接入 Mock LLM API**：调用 `POST /api/chat`
3. **实现 Tool Calling 循环**：
   - 当 LLM 返回 `tool_calls` 时，前端需要**调用对应的 API 获取数据**，再将结果发送回 LLM
   - LLM 收到工具结果后返回最终文本回复
   - 支持的工具见 `api-spec.md` 的 "Chat API" 部分
4. **展示工具调用过程**：用户能看到 "正在查询设备..." / "正在创建工单..." 等中间状态
5. **错误处理**：API 失败时给出友好提示，不能白屏或静默失败

### 模块 D：Bug 修复（必做）

附件 `buggy-component.tsx` 是一个有 **3 个 bug** 的 React 组件（设备告警面板）。

1. 找出全部 3 个 bug
2. 在你的项目里修复并集成这个组件（或单独提交修复后的文件）
3. 为每个 bug 写一段说明：**问题是什么 → 为什么是 bug → 你怎么修的**

## 5. 交付物（五项全部必交）

### 交付物 1：可运行的前端项目

- 提交完整的 git 仓库（zip 或链接均可）
- `pnpm install && pnpm dev` 即可启动（配合 mock server）
- 提交时确保 `pnpm build` 无报错

### 交付物 2：Bug 修复说明（建议 300–500 字）

每个 bug：
1. 问题现象
2. 根因分析
3. 修复方法

### 交付物 3：单元测试

- 至少覆盖 **3 个组件或函数**
- 测试框架不限（Vitest 推荐）
- 展示你认为"值得测试"的场景（不要求高覆盖率，要求测试有意义）

### 交付物 4：设计说明（建议 300–500 字）

简要说明：
- 你的项目结构和组件划分思路
- 状态管理方案选择的理由
- AI 对话模块的 Tool Calling 流程
- 你在开发中做的主要权衡取舍

### 交付物 5：AI 工具使用记录

以下任选一种形式提交：

| 形式 | 说明 |
|---|---|
| 对话日志 | 导出你与 AI 工具的对话记录（截图或文本均可） |
| 录屏片段 | 录制你使用 AI 工具开发的关键片段（2–5 分钟即可，不需要全程录） |
| 使用总结 | 500 字以内，描述你在哪些环节用了 AI、效果如何、哪里 AI 帮不上忙 |

> **为什么要交这个？** 本岗位要求"AI 工具作为日常开发的主要生产力手段"。我们想看的不是你会不会用 AI，而是**你怎么用**——怎么提问、怎么判断输出质量、怎么把 AI 输出融入你的工程。

## 7. 重要说明

### 关于 AI 工具

- **必须使用** AI 工具（Claude Code / Cursor / Copilot / ChatGPT 等均可）
- 没有 AI 使用记录的提交将被视为不完整
- 我们鼓励你大量使用 AI，但要能解释你代码的每一部分

### 关于完成度

- 4 个模块全部完成是满分目标，但如果时间不够，**优先保证完成的模块质量高**
- "3 个模块做得好" > "4 个模块都做了但很粗糙"
- 我们看的是工程素养，不只是功能清单打钩
