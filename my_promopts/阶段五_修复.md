1. 修复 Tool Calling 测试的 mock 方式
当前 ChatSidebar.test.tsx 里 query_devices 测试主要靠 callCount 区分 fetch 调用，但真实流程包含：
- POST /api/chat 第一次
- GET /api/devices?...
- POST /api/chat 第二次

请把 Tool Calling 相关测试改成按 URL + method 分支 mock，而不是只靠 callCount。至少断言：
- 第一次 /api/chat 请求 body 只包含用户消息。
- query_devices 会真正请求 /api/devices，并带上 buildingId/status/type 等参数。
- 第二次 /api/chat 请求 body 包含：
  - user 消息
  - assistant 消息，content: null，且带 tool_calls
  - tool 消息，带 tool_call_id 和 JSON.stringify 后的 content
- 最终 assistant 回复能显示在 UI 中。
注意不要依赖真实 mock server。

2. 抽出并测试 Chat API 协议重建逻辑
当前 buildApiMessages 在 ChatSidebar.tsx 内部，且没有直接测试。它是阶段五核心逻辑，建议抽到：
- src/components/chat/chatProtocol.ts
或
- src/utils/chatProtocol.ts

要求：
- 导出 buildApiMessages。
- 保持欢迎消息不发送给后端。
- 正确把 display messages 转成 ChatMessage[]。
- 对 tool-status 批次正确重建：
  - assistant tool_call message
  - 多个 tool result messages
- 给 buildApiMessages 增加独立单元测试，覆盖：
  - 普通 user + assistant 文本历史
  - 单个 tool_call
  - 多个 tool_calls
  - welcome message 被跳过

3. 修复 create_work_order 的缓存刷新条件
当前 ChatSidebar.tsx 只要 toolCalls 里存在 create_work_order 就 invalidateQueries(['workOrders'])。
请改成只有对应 create_work_order 工具执行成功时才刷新工单列表。
也就是说要同时满足：
- tc.function.name === 'create_work_order'
- 对应 results[index].success === true

- 第二次 /api/chat 后显示最终 assistant 回复。

现有 invalid JSON 测试可以保留，但请确保它断言：
- 工具状态卡片进入失败状态。
