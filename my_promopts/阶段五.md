阶段五目标：

4. 再次调用 /api/chat。
5. 展示最终 assistant 文本回复。

7. create_work_order 成功后刷新工单列表。

  - user: { role: "user", content: string }
  - assistant tool_call: { role: "assistant", content: null, tool_calls: [...] }
  - tool: { role: "tool", tool_call_id: string, content: string }

  - text message: user/assistant/content

2. 实现工具执行器
新增或提取工具执行逻辑，建议放在：
- src/components/chat/toolExecutor.ts
或
- src/utils/chatTools.ts

支持工具：
- query_devices
  - arguments: buildingId?, status?, type?
  - 调用 getDevices(params)
- query_alerts
  - arguments: buildingId?, level?
  - 调用 getAlerts(params)
- create_work_order
  - arguments: title, description, deviceId, priority
  - 调用 createWorkOrder(payload)

要求：
- function.arguments 是 JSON 字符串，必须安全解析。
- JSON 解析失败时显示工具失败状态，并给用户友好提示。
- 未知工具名要显示失败状态。
- 工具 API 调用失败要显示失败状态。
- 工具结果 content 必须 JSON.stringify 后作为 tool 消息发回 /api/chat。
- create_work_order 成功后 invalidate/refetch 工单列表 query，确保“工单管理”页能看到新工单。

3. ChatSidebar 流程
发送用户消息后：
- 先调用 sendChat(apiMessagesWithUser)
- 如果返回普通 assistant content：直接展示
- 如果返回 tool_calls：
  - 先把 assistant tool_call 消息加入 apiMessages
  - 对每个 tool_call 展示“正在调用工具”的状态卡片
  - 执行工具
  - 工具成功后更新状态卡片为成功，例如：
    - query_devices：返回 N 台设备
    - query_alerts：返回 N 条告警
    - create_work_order：创建工单 WO-xxx
  - 工具失败后更新状态卡片为失败
  - 将每个工具结果追加为 role: "tool" 消息
  - 再次调用 sendChat(fullApiMessages)
  - 展示最终 assistant content
- 如果最终回复仍返回 tool_calls，本阶段可以最多处理一轮或做有限循环保护，避免无限循环；建议最多 2 轮，并在超限时提示用户。
