# AI 工具使用记录

## 工具选择

使用 **Claude Code**（Anthropic）作为主要 AI 开发工具，分 10 个阶段完成整个项目。

## 使用方式

### 分阶段开发

每阶段遵循统一模式：
1. 向 AI 提供完整上下文：`question.md`、`attachments/wireframes.md`、`attachments/api-spec.md`、当前 `src` 代码和 `milestone/development-milestones.md`。
2. 明确要求 AI 实现具体功能范围，并列出"禁止事项"防止越界。
3. AI 完成后运行 `npm run build && npm run test` 验证。
4. 人工 review 代码，检查是否偏离里程碑、Tool Calling 协议是否正确、测试是否覆盖核心行为。

### 各阶段 AI 参与程度

| 阶段 | 内容 | AI 完成度 | 人工介入 |
|---|---|---|---|
| 1 | 工程初始化、Vite+TS+依赖+API 封装 | 高 | 确认依赖版本、tsconfig 结构 |
| 2 | 设备看板：三栏布局、楼栋选择、设备列表、统计、详情 | 高 | 确认默认楼栋选中、空状态文案 |
| 3 | 工单管理：列表、筛选、创建、状态流转 | 中 | 确认状态流转规则、Steps 逻辑 |
| 4 | AI 基础聊天：UI、消息气泡、发送/接收 | 高 | 确认消息状态提升、滚动行为 |
| 5 | AI Tool Calling 闭环：工具执行器、协议重建、状态卡片 | 中 | 协议重建逻辑（buildApiMessages）需仔细验证 |
| 6 | Bug 修复与集成：DeviceAlertPanel 的 3 个 bug | 低 | Bug 定位能力强，AI 仅需要知道方向 |
| 7 | 错误处理、空状态、边界条件 | 中 | 确认空状态区分逻辑 |
| 8 | 测试完善：6 个新测试文件 | 中 | 确认 mock 策略和断言精确性 |
| 9 | 体验打磨：样式、文案、溢出处理 | 高 | 确认不改功能 |
| 10 | 交付文档、Git 初始化 | 高 | 确认文档内容准确 |

### AI 帮不上忙的地方

- **架构边界判断**：哪些逻辑应该拆出去、哪些留在组件内，需要人对项目规模和可测试性的权衡。
- **测试可信度**：AI 可以写测试，但需要人判断测试是否真正覆盖核心行为，还是仅为凑数量。
- **交付范围控制**：AI 容易"多做"，需要人在每阶段通过"禁止事项"和 review 圈定范围。
- **视觉审美**：后台管理系统的信息密度和视觉层级，AI 的 inline style 方案够用但需要人确认整体一致性。

### 典型提问模式

```
请基于当前项目完成阶段 X 的任务。先阅读这些文件：[列表]。
需要实现的功能：[列表]。
禁止：[列表]。
验收：npm run build && npm run test。
```

### 总结

AI 极大加速了工程搭建、样板代码生成和重复性调整。但 Tool Calling 协议重建、React Query 边界行为和测试 mock 策略等环节需要人做关键决策。整体开发效率提升约 3-5 倍。
